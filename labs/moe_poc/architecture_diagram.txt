                         ZYNKCLUSTER ARCHITECTURE DIAGRAM
                     Distributed Mixture of Experts Inference
═══════════════════════════════════════════════════════════════════════════════

NOTE: Mixtral 8x7B (k=2) is used as the reference model throughout this
diagram for concreteness. The architecture applies to any open MoE model —
cluster size should match the model's k value:
  Mixtral 8x7B/8x22B: k=2  |  DeepSeek-V2: k=6  |  Grok-1: k=2
  DBRX: k=4                 |  OLMoE: k=8         |  DeepSeek-V3: k=8

OVERVIEW: Parallel Expert Execution for Lower Latency
─────────────────────────────────────────────────────────────────────────────

Traditional Approach (Petals):          ZynkCluster Approach:
Sequential Layer Distribution           Parallel Expert Distribution

  Device A (Layers 0-10)                Router selects Expert 2 + 6
       ↓ 50ms                                      ↓
  Device B (Layers 11-20)              Device A        Device B
       ↓ 50ms                           Expert 2        Expert 6
  Device C (Layers 21-31)                 ↓ 45ms         ↓ 50ms
       ↓ 50ms                                      ↓
  Total: 150ms sequential              Combine (max: 50ms)

  Problem: All devices process          Benefit: Only needed experts,
  every token sequentially              parallel execution

═══════════════════════════════════════════════════════════════════════════════
                          INFERENCE FLOW (DETAILED)
═══════════════════════════════════════════════════════════════════════════════

Step 1: User Query
──────────────────
User: "Explain quantum computing"
  ↓
Coordinator tokenizes: [1827, 15123, 19845, 17459]


Step 2: Router Selection (Sparse Activation)
─────────────────────────────────────────────
Router Model analyzes input embeddings:
  ↓
Expert Scores:
  Expert 0: 0.12  ⚫ Not selected
  Expert 1: 0.08  ⚫ Not selected
  Expert 2: 0.85  ✅ SELECTED (Weight: 0.85)
  Expert 3: 0.15  ⚫ Not selected
  Expert 4: 0.05  ⚫ Not selected
  Expert 5: 0.03  ⚫ Not selected
  Expert 6: 0.72  ✅ SELECTED (Weight: 0.72)
  Expert 7: 0.10  ⚫ Not selected

Result: Use Expert 2 + Expert 6 (sparse activation: 2/8 = 25%)


Step 3: Device Mapping
───────────────────────
Coordinator checks cluster map:
  Expert 2 → Device A (device_a_rtx3090)
  Expert 6 → Device B (device_b_rtx3060)


Step 4: Parallel Execution
───────────────────────────
┌──────────────────────────┐         ┌──────────────────────────┐
│ Thread 1: Device A       │         │ Thread 2: Device B       │
│                          │         │                          │
│ POST /expert/execute     │         │ POST /expert/execute     │
│ {                        │         │ {                        │
│   layer: 0,              │         │   layer: 0,              │
│   expert: 2,             │         │   expert: 6,             │
│   hidden_states: [...]   │         │   hidden_states: [...]   │
│ }                        │         │ }                        │
│                          │         │                          │
│ t=0ms: Request sent      │         │ t=0ms: Request sent      │
│ t=45ms: Response received│         │ t=50ms: Response received│
│                          │         │                          │
└──────────────────────────┘         └──────────────────────────┘
             │                                    │
             └─────────┬──────────────────────────┘
                       │
              Wall-clock time: 50ms
              (max of parallel operations,
               not sum of sequential)


Step 5: Combine Outputs
────────────────────────
Weighted sum:
  combined = 0.85 × Expert2_output + 0.72 × Expert6_output
  ↓
Continue to next layer...


Step 6: Complete Generation
────────────────────────────
Repeat steps 2-5 for all 32 layers, then:
  • Apply final layer normalization
  • Project to vocabulary space
  • Sample next token
  • Repeat for N tokens

Device Discovery Flow:
──────────────────────

1. Expert Node Server starts
   ↓
2. Queries local zynk_devices entry
   ↓
3. Updates: moe_capable=TRUE, hosted_experts=[...]
   ↓
4. Broadcasts availability via ZynkSync protocol
   ↓
5. Coordinator queries all paired devices
   ↓
6. Builds cluster map {device_id: {ip, experts, ...}}

                              PERFORMANCE ANALYSIS
═══════════════════════════════════════════════════════════════════════════════

Theoretical Performance (2-Device Cluster, Gigabit Ethernet):

Single Device (Baseline):
──────────────────────────
  Hardware: RTX 3090 (24GB)
  Model: Mixtral 8x7B (47B params, 13B active)
  Tokens/sec: 40
  Latency (first token): 50ms

2-Device ZynkCluster (Gigabit):
────────────────────────────────
  Device A: RTX 3090 (Experts 0-3)
  Device B: RTX 3060 (Experts 4-7)

  Tokens/sec: 30-35 (75-87% of single device)
  Latency (first token): 75ms (+25ms overhead)
  Max model size: 141B (Mixtral 8x22B)

  Trade-off: 13-25% speed loss for 3× memory capacity

Comparison with Petals (Same Hardware):
────────────────────────────────────────
  Petals (sequential):
    Latency per token: ~150ms (sum of 3 devices)
    Tokens/sec: 20-25

  ZynkCluster (parallel):
    Latency per token: ~75ms (max of 2 devices)
    Tokens/sec: 30-35

  ZynkCluster Advantage: 2× faster inference

ZynkCluster Key Points:
──────────────────────

1. Expert-Level Granularity
   • Distributes individual experts, not entire layers
   • Enables parallel execution impossible with layer-based approaches

2. Sparse Activation
   • Only 2-3 experts active per query (not all 8)
   • Reduces compute load, power consumption, network traffic

3. ZynkSync Integration
   • Reuses existing device discovery infrastructure
   • Natural extension of Zynkbot's architecture

4. Performance Advantages
   • 2-3× lower latency than sequential approaches
   • Graceful degradation when devices fail
   • Efficient resource utilization

5. Current Status
   • Design phase complete
   • Illustrative code examples provided
   • Awaiting hardware availability for testing

Hardware Requirements:
──────────────────────
  • 2+ devices with NVIDIA/AMD GPU support
  • ~13GB VRAM per node (4 experts × ~3.25GB)
  • Gigabit network between devices
  • Community collaboration welcome for testing
